① 라우터 그룹 ( ) vs 비공개 폴더 _

    (admin): 폴더를 괄호로 감싸면 URL에는 안 나타납니다. app/(admin)/dashboard/page.tsx는 그냥 /dashboard가 됩니다. 관리자 기능만 따로 묶어 관리하고 싶을 때 최고입니다.

    _components: 폴더명 앞에 언어스코어를 붙이면 Next.js가 "여기는 페이지(URL)가 아니구나"라고 인식해서 무시합니다. 해당 페이지에서만 쓰는 전용 부품을 담아둘 때 씁니다.

② 데스크탑과 모바일 분리 ((desktop) vs m/)

    주의: (desktop)은 URL에 안 남지만, m/은 URL에 /m/dashboard처럼 남게 됩니다.

    실무 조언: 반응형 디자인(Tailwind CSS의 md:, lg:)으로 하나의 코드에서 처리하는 게 유지보수 면에서 훨씬 낫습니다. 정말 UI가 180도 다르지 않다면 URL을 분리하는 방식은 비추천합니다.

③ 경로 별칭 (@/)

tsconfig.json에서 설정하는 이 기능은 필수입니다.

    기존: import Button from '../../../../components/Button' (점 찍다가 지침)

    별칭: import Button from '@/components/Button' (어디서든 한 번에 찾음)

자연스럽습니다. 현재 구조 유지하는 게 낫습니다.

  ---
store/ — 맞습니다. Zustand면 store/userStore.ts, Redux면 store/index.ts + store/slices/userSlice.ts 형태. 근데 이 프로젝트처럼
간단하면 전역 상태관리 자체가 불필요할 수 있어요.

components/common/, layout/, ui/, forms/ — 맞습니다. 다만 팀마다 다르고, shadcn/ui 같은 라이브러리 쓰면 ui/는 라이브러리가
채워줍니다. index.ts는 re-export용으로 import { Button, Icon } from "@/components/common"처럼 쓰려고 두는 겁니다.

queries/ — React Query(TanStack Query) 쓸 때 둡니다. useQuery, useMutation 래핑한 훅들이 여기 들어갑니다. 예:
queries/usePaymentQuery.ts. 지금 hooks/에 넣는 것과 비슷하지만 서버 상태 전용으로 분리하는 컨벤션입니다.

(auth)/login, (auth)/signup — 맞습니다. (admin), (customer)처럼 Route Group으로 묶고, 각각 page.tsx를 둡니다. (auth)는 레이아웃
공유(예: 로그인/회원가입 공통 배경)하거나 미들웨어에서 인증 체크 예외 처리할 때 유용합니다.
